# ВВЕДЕНИЕ

Современная авиационная отрасль предъявляет высокие требования к качеству подготовки пилотов гражданской авиации. Летные школы, осуществляющие обучение будущих пилотов, сталкиваются с необходимостью эффективного управления сложными учебными процессами, включающими теоретическую подготовку, работу на тренажерах и реальные полеты на воздушных судах.

Традиционные методы управления летной школой, основанные на бумажном документообороте и ручной обработке информации, имеют существенные недостатки: высокую трудоемкость административных операций, риск потери документов, сложность планирования ресурсов и длительность формирования отчетности для регулирующих органов.

**Актуальность темы** курсовой работы обусловлена необходимостью повышения эффективности управления летной школой за счет автоматизации основных бизнес-процессов. Внедрение автоматизированной информационной системы (АИС) позволяет оптимизировать процессы приема студентов, планирования обучения, проведения занятий, контроля успеваемости и выдачи сертификатов, что в конечном итоге способствует повышению качества подготовки пилотов и соответствию международным стандартам авиационного обучения (ИКАО).

**Цель курсовой работы** – разработка комплекса моделей автоматизированной информационной системы «Летная школа» с использованием методологий семейства IDEF и DFD для анализа и проектирования системы управления учебным процессом.

**Для достижения поставленной цели необходимо решить следующие задачи:**

1. Провести анализ предметной области и выявить основные бизнес-процессы летной школы.
2. Разработать функциональную модель (IDEF0), включающую контекстную диаграмму и необходимое количество диаграмм декомпозиции для автоматизированной и неавтоматизированной версий системы.
3. Создать модели потоков работ (IDEF3) для ключевых процессов системы.
4. Разработать модели потоков данных (DFD) в нотации Гейна-Сарсона для анализа информационных потоков.
5. Расширить информационную модель (IDEF1X) дополнительной сущностью с необходимыми атрибутами.
6. Провести сравнительный анализ автоматизированной и неавтоматизированной версий системы.
7. Оценить экономическую эффективность внедрения АИС.

# 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

## 1.1. Общая характеристика предметной области

Летная школа представляет собой специализированное образовательное учреждение, осуществляющее подготовку пилотов гражданской авиации по программам различных уровней квалификации:
- **PPL** (Private Pilot License) – лицензия частного пилота
- **CPL** (Commercial Pilot License) – лицензия коммерческого пилота
- **ATPL** (Airline Transport Pilot License) – лицензия линейного пилота

Деятельность летной школы регулируется Федеральным агентством воздушного транспорта (Росавиация) и должна соответствовать требованиям Международной организации гражданской авиации (ИКАО), в частности, Приложению 1 к Конвенции о международной гражданской авиации.

**Основные ресурсы летной школы:**

- **Человеческие ресурсы:**
  - Инструкторы по теоретической подготовке
  - Инструкторы по полетам (имеющие сертификат FI – Flight Instructor)
  - Экзаменаторы
  - Административный персонал
  - Технический персонал (механики, техники)

- **Материально-технические ресурсы:**
  - Учебные воздушные суда (одно- и многодвигательные самолеты)
  - Тренажеры и симуляторы полета
  - Учебные классы и аудитории
  - Аэродром с необходимой инфраструктурой
  - Техническая база для обслуживания воздушных судов

- **Информационные ресурсы:**
  - Учебные планы и программы
  - Методические материалы
  - Бортовые журналы воздушных судов
  - Личные дела студентов
  - Журналы учета налета часов
  - Документация по техническому обслуживанию
  - Отчетность для регулирующих органов

**Внешние субъекты взаимодействия:**
- Кандидаты на обучение (будущие студенты)
- Росавиация (предоставление отчетности, получение лицензий)
- Медицинские учреждения (ВЛЭК – врачебно-летная экспертная комиссия)
- Авиакомпании (потенциальные работодатели выпускников)

Процесс обучения в летной школе является длительным и многоэтапным, требующим координации множества ресурсов и строгого соблюдения нормативных требований безопасности полетов.

## 1.2. Основные бизнес-процессы летной школы

Анализ деятельности летной школы позволяет выделить пять основных бизнес-процессов, охватывающих полный цикл подготовки пилота:

**1. Прием и регистрация студентов**

Данный процесс включает следующие этапы:
- Прием заявлений от кандидатов на обучение
- Проверку предоставленных документов (паспорт, документ об образовании)
- Проверку соответствия требованиям (возраст, уровень образования, знание языков)
- Медицинское освидетельствование в ВЛЭК для получения авиационного медицинского сертификата
- Заключение договора на обучение
- Оформление зачисления студента в школу

**Входы:** Заявления от кандидатов, документы (паспорт, аттестат/диплом), медицинские справки

**Выходы:** Зачисленные студенты, личные дела, договоры

**Проблемы неавтоматизированной версии:**
- Длительность обработки документов (3-5 дней)
- Риск потери документов
- Сложность проверки комплектности документов
- Отсутствие оперативной информации о статусе заявления

**2. Планирование обучения**

Процесс планирования включает:
- Формирование учебных групп с учетом уровня подготовки
- Составление расписания теоретических занятий
- Планирование занятий на тренажерах
- Резервирование воздушных судов для полетных занятий
- Назначение инструкторов с учетом их квалификации и загрузки
- Учет метеоусловий и технического состояния самолетов

**Входы:** Данные о зачисленных студентах, учебные планы, доступность ресурсов

**Выходы:** Расписание занятий, распределение ресурсов, назначения инструкторов

**Проблемы неавтоматизированной версии:**
- Высокая трудоемкость ручного планирования
- Конфликты при распределении ресурсов
- Сложность оперативной корректировки расписания
- Невозможность быстрой оптимизации использования ресурсов

**3. Проведение занятий**

Основной процесс обучения, включающий:
- Проведение теоретических занятий по дисциплинам:
  - Аэродинамика и динамика полета
  - Авиационная метеорология
  - Навигация и радионавигация
  - Авиационное право и правила полетов
  - Человеческий фактор
- Проведение тренажерных занятий на симуляторах
- Проведение полетных занятий:
  - Предполетная подготовка и брифинг
  - Предполетный осмотр воздушного судна
  - Выполнение полета под контролем инструктора
  - Самостоятельные полеты (solo)
  - Послеполетный разбор и дебрифинг
- Фиксация результатов занятий

**Входы:** Расписание, учебные материалы, воздушные суда, студенты

**Выходы:** Отметки о посещаемости, оценки, записи в бортовых журналах, налет часов

**Проблемы неавтоматизированной версии:**
- Ручное ведение журналов учета
- Риск ошибок при подсчете налета часов
- Сложность контроля выполнения учебного плана
- Отсутствие оперативной информации о прогрессе студента

**4. Контроль успеваемости**

Процесс оценки знаний и навыков студентов:
- Текущий контроль успеваемости по теоретическим дисциплинам
- Проведение письменных тестов
- Устные экзамены
- Практические проверки навыков пилотирования (чек-райды)
- Анализ готовности к сертификации
- Формирование итоговых оценок

**Входы:** Результаты занятий, данные о налете часов, текущие оценки

**Выходы:** Результаты экзаменов, допуск к сертификации, рекомендации

**Проблемы неавтоматизированной версии:**
- Ручной подсчет баллов и анализ результатов
- Сложность выявления слабых мест в подготовке
- Длительность формирования отчетов об успеваемости

**5. Выдача сертификатов**

Заключительный процесс:
- Проверка выполнения всех требований программы обучения
- Оформление сертификата пилота установленного образца
- Регистрация сертификата в базе Росавиации
- Выдача сертификата и документов об образовании выпускнику
- Формирование отчетности для регулирующих органов

**Входы:** Списки успешно завершивших обучение, результаты экзаменов

**Выходы:** Сертификаты пилотов, документы об образовании, отчеты для Росавиации

**Проблемы неавтоматизированной версии:**
- Ручное оформление документов
- Риск ошибок в персональных данных
- Длительность формирования отчетности



## 1.3. Цели и задачи автоматизации

**Основная цель автоматизации** – повышение эффективности управления летной школой за счет:
- Сокращения времени выполнения административных операций
- Повышения точности и достоверности данных
- Улучшения качества планирования и использования ресурсов
- Обеспечения оперативного доступа к информации
- Автоматизации формирования отчетности

**Задачи автоматизации:**

1. **Автоматизация документооборота:**
   - Электронная регистрация заявлений и документов студентов
   - Создание электронных личных дел
   - Формирование договоров и документов об образовании
   - Электронная подпись документов

2. **Автоматизация планирования:**
   - Автоматическое формирование расписания с учетом ограничений
   - Оптимизация распределения инструкторов и воздушных судов
   - Учет метеоусловий и технического состояния самолетов
   - Оперативная корректировка расписания

3. **Автоматизация учета учебного процесса:**
   - Электронный журнал посещаемости
   - Автоматический расчет налета часов
   - Интеграция с GPS-данными полетов

# 2. ФУНКЦИОНАЛЬНАЯ МОДЕЛЬ (IDEF0)

Функциональная модель разработана в соответствии с методологией IDEF0 и стандартом Р 50.1.028-2001. Модель создана в двух версиях: автоматизированной (с применением АИС «Летная школа») и неавтоматизированной (традиционный подход с бумажным документооборотом).

## 2.1. Контекстная диаграмма A-0

Контекстная диаграмма A-0 представляет систему управления летной школой как единое целое, взаимодействующее с внешней средой.

**Цель моделирования:** Разработка функциональной модели процессов управления летной школой для автоматизации деятельности образовательного учреждения по подготовке пилотов гражданской авиации.

**Точка зрения:** Модель разработана с точки зрения руководителя летной школы, который отвечает за организацию учебного процесса, контроль качества обучения, управление ресурсами и соблюдение авиационных стандартов.

### Автоматизированная версия (диаграмма A-0-автомат)

**Входы:**
1. Онлайн-заявки – электронные заявления от кандидатов через веб-портал
2. Электронные документы – сканированные копии документов об образовании, медицинских справок
3. Требования властей – нормативные документы от Росавиации и ИКАО в электронном виде

**Управление:**
1. Правила авиации – законодательные акты, регулирующие гражданскую авиацию (ФАП, ИКАО Annex 1)
2. Стандарты обучения – утвержденные учебные программы (PPL, CPL, ATPL)
3. Требования безопасности – правила безопасности полетов

**Выходы:**
1. Квалифицированные пилоты – выпускники, прошедшие полный курс обучения
2. Электронные сертификаты – цифровые документы с электронной подписью
3. Автоматические отчеты – статистика и отчетность, генерируемая АИС

**Механизмы:**
1. АИС «Летная школа» – автоматизированная информационная система
2. Инструкторы – квалифицированный преподавательский состав
3. Самолеты – учебная авиационная техника с современным оборудованием
4. Персонал – административный и технический персонал

### Неавтоматизированная версия (диаграмма A-0-ручная)

**Входы:**
1. Заявки на обучение – бумажные заявления, подаваемые лично или по почте
2. Документы студентов – оригиналы или заверенные копии
3. Требования властей – бумажные копии нормативных актов

**Управление:**
1. Правила авиации – печатные версии законодательных актов
2. Стандарты обучения – бумажные учебные программы
3. Требования безопасности – инструкции в печатном виде

**Выходы:**
1. Квалифицированные пилоты – выпускники с дипломами
2. Сертификаты – бумажные документы с печатями и подписями
3. Отчеты о деятельности – рукописные или напечатанные отчеты

**Механизмы:**
1. Инструкторы – преподавательский состав
2. Самолеты – учебные воздушные суда
3. Учебные классы – аудитории с досками и пособиями
4. Персонал – секретари, методисты, администраторы

## 2.2. Диаграмма декомпозиции A0

Контекстный процесс «Управление летной школой» декомпозируется на пять последовательных процессов:

# 3. МОДЕЛЬ ПОТОКОВ РАБОТ (IDEF3)

Методология IDEF3 используется для детального описания последовательности действий в ключевых процессах летной школы.

## 3.1. IDEF3-диаграмма «Процесс проведения полетного занятия»

Данная диаграмма описывает процесс проведения одного полетного занятия от предполетной подготовки до фиксации результатов.

**Последовательность действий:**

**UOB 1: Предполетный брифинг (30 мин)**
- Обсуждение целей полета
- Разбор маршрута и метеоусловий
- Инструктаж по безопасности

→ **Переход к UOB 2**

**UOB 2: Предполетный осмотр воздушного судна (15-20 мин)**
- Внешний осмотр самолета
- Проверка уровня топлива и масла
- Проверка системы управления

→ **Решение: Самолет исправен?**
- Да → UOB 3
- Нет → **Завершение занятия / Замена самолета**

**UOB 3: Выполнение полета (60-90 мин)**
- Запуск двигателя
- Руление к взлетной полосе
- Взлет и набор высоты
- Выполнение учебных упражнений
- Заход на посадку и посадка
- Руление на стоянку

→ **Переход к UOB 4**

**UOB 4: Послеполетный разбор (30 мин)**
- Анализ выполнения полета
- Обсуждение ошибок и успехов
- Рекомендации инструктора

→ **Переход к UOB 5**

**UOB 5: Фиксация результатов (10-15 мин)**
- Запись в бортовой журнал самолета
- Запись в личный журнал студента
- Внесение данных о налете часов
- Выставление оценки

→ **Конец процесса**

**Общая длительность:** 3-5 часов (включая подготовку и оформление)

**Особенности автоматизированной версии:**
- Электронный брифинг с использованием планшетов
- GPS-запись трека полета
- Автоматическая фиксация времени полета
- Электронная запись результатов в АИС

## 3.2. IDEF3-диаграмма «Процесс регистрации студента»

Описывает процесс от подачи заявления до зачисления в школу.

**Последовательность действий:**

**UOB 1: Подача заявления (1 час)**
- Онлайн-заполнение анкеты / Личная подача документов
- Загрузка сканов документов / Предоставление оригиналов
- Выбор программы обучения

→ **Переход к UOB 2**

**UOB 2: Первичная проверка документов (2-4 часа)**
- Проверка комплектности документов
- Проверка соответствия требованиям

→ **Решение: Документы в порядке?**
- Нет → **Запрос дополнительных документов** → возврат к UOB 1
- Да → **Параллельный переход к UOB 3 и UOB 4**

**Параллельные процессы:**

**UOB 3: Проверка образования (1 день)** 
- Верификация диплома/аттестата
- Проверка оценок и профиля

**И (одновременно)**

**UOB 4: Медицинское освидетельствование (3-5 дней)**
- Направление в ВЛЭК
- Прохождение медкомиссии
- Получение медицинского сертификата

**Синхронизация:** Оба процесса должны завершиться

→ **Решение: Все проверки пройдены?**
- Нет → **Отказ в зачислении** → Конец
- Да → UOB 5

**UOB 5: Заключение договора (1-2 часа)**
- Формирование договора
- Подписание договора
- Оплата (или согласование рассрочки)

→ **Переход к UOB 6**

**UOB 6: Зачисление (30 мин)**
- Создание личного дела студента
- Присвоение учебного номера
- Внесение в базу данных школы
- Формирование студенческого билета

→ **Конец процесса**

**Общая длительность:** 1-2 недели

**Особенности автоматизированной версии:**
- Автоматическая проверка комплектности
- Электронная подпись договора

# 4. МОДЕЛЬ ПОТОКОВ ДАННЫХ (DFD)

Диаграммы потоков данных созданы в нотации Гейна-Сарсона для процесса A3 «Проведение занятий».

## 4.1. DFD-диаграмма процесса A3 (автоматизированная версия)

**Внешние сущности:**
1. Студент
2. Инструктор
3. Росавиация (регулятор)

**Процессы:**

**3.1 Проведение теоретических занятий**
- Входы: Расписание [Д1], Учебные материалы [Д2]
- Выходы: Результаты теории → Д5

**3.2 Проведение тренажерных занятий**
- Входы: Расписание [Д1]
- Выходы: Результаты тренажера → Д5

**3.3 Проведение полетных занятий**
- Входы: Расписание [Д1], Данные самолета [Д4]
- Выходы: Результаты полетов → Д5, GPS-данные → Д6

**3.4 Фиксация результатов**
- Входы: Результаты теории/тренажера/полетов [Д5], GPS-данные [Д6]
- Выходы: Электронный журнал → Д7, Отчеты → Росавиация

**Хранилища данных:**

- **Д1: Расписание** – электронное расписание занятий
- **Д2: Учебные материалы** – электронная библиотека
- **Д3: Личные дела студентов** – база данных студентов
- **Д4: База данных самолетов** – информация о ВС
- **Д5: Промежуточные результаты** – временное хранилище
- **Д6: GPS-треки полетов** – записи траекторий
- **Д7: Электронный журнал занятий** – основная база результатов

**Потоки данных:**

От Студента: Запрос расписания → 3.1, 3.2, 3.3
От Инструктора: Оценки → 3.1, 3.2, 3.3
К Студенту: Результаты обучения ← 3.4
К Инструктору: Статистика группы ← 3.4
К Росавиации: Отчеты о налете ← 3.4

## 4.2. DFD-диаграмма процесса A3 (неавтоматизированная версия)

**Внешние сущности:**
1. Студент
2. Инструктор
3. Росавиация

**Процессы:**

**3.1 Проведение теоретических занятий**
- Входы: Бумажное расписание [Д1], Учебники [Д2]
- Выходы: Оценки на бумаге → Д5

**3.2 Проведение тренажерных занятий**
- Входы: Бумажное расписание [Д1]
- Выходы: Оценки → Д5

**3.3 Проведение полетных занятий**
- Входы: Бумажное расписание [Д1], Бортжурнал [Д4]
- Выходы: Записи в бортжурнале → Д4, Оценки → Д5

**3.4 Фиксация результатов**
- Входы: Оценки [Д5], Записи из бортжурналов [Д4]
- Выходы: Журнал посещаемости → Д7, Рукописные отчеты → Росавиация

**Хранилища данных:**

- **Д1: Доска объявлений** – расписание на стенде
- **Д2: Библиотека** – бумажные учебники
- **Д3: Картотека студентов** – личные дела в папках
- **Д4: Бортовые журналы** – журналы учета полетов ВС
- **Д5: Листы с оценками** – бумажные оценочные листы
- **Д6: (отсутствует)** – нет GPS-данных
- **Д7: Журнал посещаемости** – бумажный журнал

**Потоки данных:**

От Студента: Устный запрос → 3.1, 3.2, 3.3
От Инструктора: Рукописные оценки → 3.1, 3.2, 3.3
К Студенту: Устное сообщение результатов ← 3.4
К Инструктору: Копии страниц журнала ← 3.4
К Росавиации: Ежемесячные отчеты на бумаге ← 3.4

# 5. ИНФОРМАЦИОННАЯ МОДЕЛЬ (IDEF1X)

## 5.1. Описание информационной модели

Информационная модель разработана в методологии IDEF1X и включает основные сущности для управления летной школой:

**Основные сущности:**

1. **Student** – Студенты
   - StudentID (PK)
   - FirstName, LastName
   - DateOfBirth
   - MedicalCertificate
   - EnrollmentDate

2. **Instructor** – Инструкторы
   - InstructorID (PK)
   - FirstName, LastName
   - LicenseNumber
   - CertificateExpiry

3. **Aircraft** – Воздушные суда
   - AircraftID (PK)
   - RegistrationNumber
   - Model, Manufacturer
   - TotalFlightHours
   - LastMaintenanceDate

4. **Course** – Учебные программы
   - CourseID (PK)
   - CourseName (PPL, CPL, ATPL)
   - Duration
   - TheoryHours, FlightHours

5. **Lesson** – Занятия
   - LessonID (PK)
   - StudentID (FK)
   - InstructorID (FK)
   - AircraftID (FK)
   - LessonDate, Duration
   - LessonType (Theory/Simulator/Flight)
   - Grade

6. **Exam** – Экзамены
   - ExamID (PK)
   - StudentID (FK)
   - ExamDate
   - ExamType
   - Score, Result

7. **Certificate** – Сертификаты
   - CertificateID (PK)
   - StudentID (FK)
   - CourseID (FK)
   - IssueDate
   - CertificateNumber

## 5.2. Дополнительная сущность MaintenanceLog

В рамках курсовой работы добавлена сущность **MaintenanceLog** для учета технического обслуживания воздушных судов.

**Атрибуты сущности MaintenanceLog:**

1. **MaintenanceLogID** (PK, INT) – Первичный ключ
2. **AircraftID** (FK, INT) – Внешний ключ к Aircraft
3. **MaintenanceDate** (DATETIME) – Дата обслуживания
4. **MaintenanceType** (VARCHAR(100)) – Тип обслуживания (плановое/внеплановое)
5. **Description** (TEXT) – Описание выполненных работ
6. **Cost** (DECIMAL(10,2)) – Стоимость обслуживания
7. **MechanicName** (VARCHAR(200)) – ФИО механика
8. **Notes** (TEXT, nullable) – Дополнительные примечания

**Связи:**

Сущность MaintenanceLog связана с Aircraft отношением один-ко-многим (1:N):
- Один Aircraft может иметь множество записей MaintenanceLog
- Каждая запись MaintenanceLog относится к одному Aircraft
- При удалении Aircraft удаление связанных записей MaintenanceLog запрещено (RESTRICT)

**Назначение:**

MaintenanceLog обеспечивает:
1. Отслеживание истории обслуживания каждого ВС
2. Планирование планового ТО
3. Контроль затрат на эксплуатацию
4. Соблюдение требований безопасности полетов
5. Формирование отчетности для регулирующих органов

## 5.3. Интеграция в базу данных

**Файл MaintenanceLog.cs:**

```csharp
public class MaintenanceLog
{
    public int MaintenanceLogID { get; set; }  // PK
    public int AircraftID { get; set; }        // FK
    public DateTime MaintenanceDate { get; set; }
    public string MaintenanceType { get; set; }
    public string Description { get; set; }
    public decimal? Cost { get; set; }
    public string MechanicName { get; set; }
    public string? Notes { get; set; }
    
    // Navigation property
    public virtual Aircraft Aircraft { get; set; }
}
```

**Обновление FlightSchoolDbContext.cs:**

```csharp
public DbSet<MaintenanceLog> MaintenanceLogs { get; set; }

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<MaintenanceLog>()
        .HasOne(m => m.Aircraft)
        .WithMany(a => a.MaintenanceLogs)
        .HasForeignKey(m => m.AircraftID)
        .OnDelete(DeleteBehavior.Restrict);
}
```

**Обновление Aircraft.cs:**

```csharp
public class Aircraft
{
    // ... existing properties ...
    
    public virtual ICollection<MaintenanceLog> MaintenanceLogs { get; set; }
}
```

**SQL-запрос для создания таблицы:**

```sql
CREATE TABLE MaintenanceLog (
    MaintenanceLogID INT PRIMARY KEY IDENTITY(1,1),
    AircraftID INT NOT NULL,
    MaintenanceDate DATETIME NOT NULL,
    MaintenanceType VARCHAR(100) NOT NULL,
    Description TEXT NOT NULL,
    Cost DECIMAL(10,2),
    MechanicName VARCHAR(200) NOT NULL,
    Notes TEXT,
    FOREIGN KEY (AircraftID) REFERENCES Aircraft(AircraftID)
        ON DELETE RESTRICT
);
```

   - Сокращение времени на административные операции на 60-70%
